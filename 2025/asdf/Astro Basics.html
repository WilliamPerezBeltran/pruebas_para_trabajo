Prueba Técnica: Desarrollo Full Stack (Angular + Ruby o Nest.js)

1. Introducción

El objetivo de esta prueba técnica es evaluar tus conocimientos en el desarrollo de aplicaciones front-end con Angular y back-end (usando NestJS o Ruby on Rails) integradas mediante una API RESTful. Se pondrá especial énfasis en la gestión de autenticación basada en tokens, consumo de servicios utilizando promesas (evitando el uso directo de .subscribe() en favor de patrones basados en async/await o .toPromise()), manejo de rutas y guards dinámicos, validación y gestión de formularios reactivos, manejo de errores y estructuración de una base de datos SQL con relaciones entre entidades.

[!IMPORTANT]
En el uso o llamado de servicios http en el sistema frontend, es importante evitar el uso del llamado de las fuciones directamente en el componente, es decir, evitar tener llamados de servicios de la siguiente manera, no obstante no es obligatorio, pero se tendra en cuenta:

export class MyComponent()
  private readonly myService = @inject(MyService)

  constructor()
	  this.getAll()
  

  private getAll()
	  this.myService.getAll().pipe().subscribe(
		  next: (response:<T>)=>
			  // Hacer algo
		  ,
		  error: (error:<T>)=>
			  // manejo de errores
		  ,
	  )

la idea con esto es poder ver el metodo de abstraccion de funcionalidades para factorizar el codigo es decir, se espera que se maneje algo por el estilo

import MyRequests from './my-request.service'

export class MyComponent()
  private readonly MyRequests = @inject(MyRequests)

  constructor()
	  this.MyRequests.getAll().then((respone)=>
		  //algo
	  ).catch((error)=>
		  //manejo de erores
	  )

La idea del ejercicio, el poder llegar a construir servicios que puedan ser inhjectados en una fucnoin que valide estados, errores, respuesteas y evitar errores de bloqueo sel componente por posibles fallos de conexion al servicio, adicionalmente se valorara el uso de herecia entre el aplicativo.

2. Alcance y Requerimientos Generales

  • Tiempo estimado: 1 a 3 días.
  • Estructura del proyecto:
    • Frontend: Aplicación Angular.
    • Backend: API RESTful desarrollada en NestJS o Ruby on Rails.
    • Base de datos: SQL (PostgreSQL, MySQL, etc.).
  • Autenticación: Se debe implementar un sistema basado en tokens (por ejemplo, JWT) que valide en cada llamada la identidad del usuario y determine permisos/roles (Ideal, en los guards, tener el uso de un servicio que valide si el token caduco, es incorrecto, etc).
  • Estilos: Uso de un framework de CSS (por ejemplo, Bootstrap, Tailwind o Material Design) para lograr un diseño responsivo y moderno.

3. Requerimientos Funcionales y Técnicos

3.1. Frontend (Angular)

  • Estructura y Consumo de Servicios:

    • La aplicación debe consumir la API RESTful usando servicios que retornen promesas (o que permitan el uso de async/await), evitando el patrón directo de .subscribe() en componentes. Esto es clave para evitar bloqueos en el flujo y facilitar el encadenamiento de peticiones, por ejemplo, cargar datos de un país específico antes de renderizar la vista.

  • Rutas y Guards Dinámicos:

    • Configuración de rutas en Angular con guards que, de forma dinámica, verifiquen roles y permisos en función del token del usuario.

  • Gestión de Errores:

    • Implementar un manejo global de errores en las peticiones HTTP (mediante interceptores) y en los formularios.

  • Formularios Reactivos:

    • Utilizar Reactive Forms para la captura y validación de datos. Se deben incluir validaciones en tiempo real y mostrar mensajes de error adecuados.

  • Integración de Autenticación:

    • Cada llamada a la API debe enviar el token de autenticación (por ejemplo, mediante un interceptor HTTP).

  • Uso de CSS Framework:

    • Incorporar un framework de CSS para un diseño limpio, responsivo y acorde a estándares modernos.

3.2. Backend (NestJS o Ruby on Rails)

  • API RESTful:

    • Diseñar y desarrollar una API que gestione el CRUD para las siguientes entidades:
      • Compañías: Con los siguientes campos:
        • Nombre
        • Sector
        • Ciudad (relación con la tabla de ciudades)
        • Departamento (relación con la tabla de departamentos)
        • Producto(s) (relación con la tabla de productos)
        • Teléfono
        • Dirección
        • Activos y pasivos
        • Usuarios asociados (relación con la tabla de usuarios)
      • Productos:
        • Nombre
        • Categoría
        • Precio
      • Usuarios:
        • Nombre
        • Apellido
        • Cargo
        • Salario
        • Teléfono
        • Correo

  • Base de Datos SQL:

    • Se debe diseñar la siguiente estructura:
      • Tabla de Departamentos:
        • Estos datos se proporcionarán en formato JSON y serán cargados inicialmente (no requieren CRUD).
      • Tabla de Ciudades y/o Municipios:
        • Debe contener una referencia (llave foránea) al Departamento correspondiente.
      • Tabla de Compañías:
        • Debe estar asociada a la tabla de ciudades (o departamentos) mediante llaves foráneas.

  • Relaciones:

    • Una compañía tendrá:
      • Una relación uno a muchos con los usuarios registrados.
      • Una relación (posiblemente uno a muchos) con productos (dependiendo del diseño, puede ser que una compañía ofrezca varios productos).

  • Autenticación y Seguridad:

    • Implementar un sistema de autenticación (por ejemplo, JWT) que verifique el token en cada llamada.
    • Definir roles y permisos, de manera que la API valide si el usuario tiene permisos para realizar la acción solicitada.

4. Flujo y Detalles de Implementación

4.1. Configuración del Proyecto

  • Repositorio:
    • Crear un repositorio (GitHub, GitLab, etc.) que contenga dos carpetas principales: frontend y backend.
  • Documentación:
    • Incluir un archivo README.md con instrucciones detalladas para levantar ambos entornos (incluyendo variables de entorno, comandos de migración y seed de la base de datos).

4.2. Implementación en el Frontend

  • Servicios y Consumo de API:

    • Crear servicios que encapsulen las llamadas a la API.
    • Convertir las respuestas a promesas y utilizar async/await en los componentes o funciones donde se requiera la sincronización de peticiones.

  • Rutas y Guards:

    • Configurar el módulo de rutas (AppRoutingModule) con rutas protegidas mediante guards dinámicos basados en roles.

  • Interceptors:

    • Configurar un interceptor HTTP que:
      • Inyecte el token en cada solicitud.
      • Capture errores globalmente y redirija a páginas de error o login si es necesario.

  • Componentes CRUD:

    • Implementar vistas para gestionar las entidades:
      • Listado, creación, edición y eliminación de compañías, productos y usuarios.
    • Utilizar formularios reactivos con validaciones en cada uno de estos componentes.

  • Diseño:

    • Aplicar estilos mediante un framework de CSS para lograr un diseño moderno y responsivo.

4.3. Implementación en el Backend

  • Endpoints CRUD:

    • Desarrollar endpoints para el CRUD de compañías, productos y usuarios.
    • Proteger cada endpoint con middleware o filtros de autenticación que verifiquen el token.

  • Gestión de la Base de Datos:

    • Diseñar la base de datos con las tablas y relaciones indicadas.
    • Crear migraciones y seeders para cargar los datos iniciales de departamentos y municipios (a partir del JSON proporcionado).

  • Autenticación:

    • Implementar la lógica de autenticación y autorización usando JWT (o la metodología que prefieras).
    • Asegurar que en cada llamada a la API se valide el token y se verifiquen los permisos correspondientes.

5. Pruebas y Validación

  • Documentación de Endpoints:

    • Se recomienda utilizar herramientas como Postman o Swagger para documentar y probar los endpoints de la API.

  • Pruebas Unitarias / Integración (Opcional):

    • Si el tiempo lo permite, incluir pruebas unitarias o de integración tanto en el frontend como en el backend.

  • Validación de Errores:

    • Asegurarse de que tanto el frontend como el backend manejen adecuadamente los errores, proporcionando feedback claro al usuario.

6. Entregables y Criterios de Evaluación

6.1. Entregables

  • Código fuente:

    • Repositorio con el código organizado en dos secciones: frontend, backend, Scripts de creación de la base de datos (De no haber usado algun ORM).

  • Documentación:

    • README.md con instrucciones para la instalación, configuración y ejecución del proyecto.
    • Documentación de los endpoints (por ejemplo, con Swagger o Postman).

  • Demo (obligatorio):

    • Un demo o enlace a una versión desplegada de la aplicación pueden usar Vercel, Netlify, Railway, entre otros, para la base de datos, pueden usar Supabase, Turso, Neon, entre otros, adiconal a esto se requiere el acceso a la base de datos. Tener presente que el no tener el aplicativo desplegado para sus pruebas, la misma no sera tenida en cuenta.

6.2. Criterios de Evaluación

  • Calidad del Código:

    • Claridad, estructura y comentarios adecuados.

  • Cumplimiento de Requerimientos:

    • Correcta implementación de la autenticación, manejo de tokens y roles.
    • Consumo de servicios mediante promesas en lugar de .subscribe().
    • Implementación correcta de rutas, guards y formularios reactivos.

  • Integración Frontend/Backend:

    • Flujo correcto de autenticación y validación de permisos.
    • Manejo adecuado de relaciones en la base de datos (ciudades, departamentos, compañías, productos y usuarios).

  • Diseño y Usabilidad:

    • Interfaz moderna y responsiva utilizando un framework de CSS.

  • Documentación y Pruebas:

    • Claridad en las instrucciones y documentación de endpoints.
    • Inclusión de pruebas o validaciones de error.

7. Conclusiones

Esta prueba técnica busca evaluar tu capacidad para desarrollar una aplicación full stack que integre de forma robusta el frontend en Angular con un backend basado en una API RESTful. Se valorará la correcta estructuración del código, el manejo adecuado de la autenticación y la integración entre servicios, así como la aplicación de buenas prácticas de desarrollo y un diseño de interfaz de usuario moderno.

Recuerda documentar tus decisiones, comentar el código y priorizar la funcionalidad esencial en caso de limitaciones de tiempo.
